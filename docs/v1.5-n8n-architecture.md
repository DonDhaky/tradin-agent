# V1.5 Architecture Note: Firestore -> n8n Enrichment

## Context (Current Project)
- Trade-in requests are created in `app/dashboard/actions.ts` and persisted via `lib/firestore/trade-in-requests.ts`.
- Firestore is already the core system of record.
- Next.js app is intentionally lean and should stay focused on auth, intake, and dashboard UX.

## 1) Cleanest Trigger Strategy
- Trigger on Firestore document creation: `tradeInRequests/{requestId}`.
- Use an event-driven backend trigger, not a direct call from the Next.js server action.
- Add event fields at creation time for workflow tracking:
  - `workflowStatus`: `pending | processing | completed | failed`
  - `estimationVersion`: string (for future model/workflow versions)
  - `estimationUpdatedAt`: timestamp nullable

Why this is clean:
- Request persistence succeeds independently from automation.
- No extra latency on the user submit path.
- Firebase remains source of truth; automation is downstream.

## 2) Should Cloud Functions Sit Between Firestore and n8n?
- Yes, for V1.5 the cleanest boundary is: Firestore onCreate -> Firebase Cloud Function -> n8n webhook.

Rationale:
- Keeps webhook secret and signing logic off the web app.
- Provides a controlled, auditable integration boundary.
- Allows retries/idempotency handling without coupling to UI code.
- Avoids exposing n8n directly as a trusted writer from the browser path.

## 3) Payload to Send to n8n
Send a minimal, explicit, versioned payload from Cloud Function to n8n:

```json
{
  "eventType": "trade_in_request.created",
  "eventVersion": "1.0",
  "eventId": "<uuid>",
  "occurredAt": "<iso_timestamp>",
  "request": {
    "id": "<firestore_doc_id>",
    "userId": "<uid>",
    "userEmail": "<email>",
    "category": "<category>",
    "brand": "<brand>",
    "model": "<model>",
    "storage": "<storage>",
    "condition": "<condition>",
    "accessories": "<accessories>",
    "notes": "<notes>",
    "expectedPrice": 450,
    "createdAt": "<iso_timestamp>"
  }
}
```

Also send auth headers:
- `X-Signature` (HMAC SHA-256 over raw body with shared secret)
- `X-Timestamp`

## 4) n8n Workflow (Step-by-Step)
1. Webhook receives event.
2. Verify signature/timestamp and reject invalid requests.
3. Idempotency check using `eventId` (skip duplicates).
4. Mark Firestore request `workflowStatus = processing`.
5. Run lightweight enrichment (price references, condition-based adjustment heuristics, confidence score).
6. Build estimation summary text (range, rationale bullets, confidence).
7. Send estimation summary email to `userEmail`.
8. Write results back to Firestore under the request document:
   - `estimation`: `{ low, high, suggested, confidence, summary, sources, generatedAt }`
   - `workflowStatus = completed`
   - `estimationUpdatedAt`
9. On error: write `workflowStatus = failed` and `workflowError` (sanitized), with retry policy.

## 5) What Stays in Web App vs Externalized

Web app keeps:
- Auth/session handling.
- Trade-in intake form submission.
- Firestore write as primary creation action.
- Dashboard rendering of request + estimation fields.

Externalize to Cloud Functions + n8n:
- Event trigger and webhook dispatch.
- Enrichment/market lookup logic.
- Estimation synthesis.
- Email delivery orchestration.
- Retry/backoff and integration observability.

## 6) Defer to Later Version
- Full marketplace scraping pipelines and high-frequency repricing.
- Multi-channel notifications (Telegram, Sheets, CRM sync).
- Human-in-the-loop admin review queues and SLA workflows.
- Advanced model-based valuation and A/B strategy tuning.
- Complex event bus/queue infra unless volume justifies it.

## Recommended V1.5 Flow
`Next.js submit -> Firestore create -> Cloud Function onCreate -> n8n webhook -> enrichment + email -> Firestore update`
